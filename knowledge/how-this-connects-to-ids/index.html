Post 04 · Checkability

How does this connect to IDS?

IDS is most useful when it is not the starting point. It should encode information needs that have already been justified by use cases and processes.

From intent to machine-checkable rule

A requirement such as “maintainable assets need a unique identifier” must be translated into a structured specification: applicable entity, classification, required property, data type, allowed value and delivery milestone.

Use case → Information need → IFC entity/property → IDS specification → Model check

What IDS does well

  • Makes selected IFC requirements explicit and portable.
  • Supports repeatable validation across software.
  • Separates checking logic from long narrative documents.
  • Creates traceable evidence for acceptance.
Important: IDS can verify whether information is present and structured correctly. It does not decide whether the requested information is useful. That decision belongs to the use-case and process layer.

Connected rather than isolated

The IPM model links each IDS rule back to its reason, responsible process, source role and expected output. A failed rule can therefore be understood as a process issue, not only a technical error.
